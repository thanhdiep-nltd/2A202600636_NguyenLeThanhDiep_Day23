## 2. Architecture

The system is designed as a StateGraph support-ticket routing system with human-in-the-loop, error-handling retry loops, and LLM-based query classification.

### Nodes:
- `intake`: Preprocesses and normalizes the input query.
- `classify`: LLM structured classification of query intent (simple, tool, missing_info, risky, error) with priority.
- `tool`: Simulated tool execution representing database queries or order status updates.
- `evaluate`: Evaluation node checking if tool output contains any issues (via LLM-as-judge or heuristic).
- `answer`: LLM-generated polite and helpful grounded response.
- `clarify`: Requests missing details for vague queries.
- `risky_action`: Prepares sensitive actions before approval.
- `approval`: Human-in-the-loop checkpoint, allowing manual intervention.
- `retry`: Increments retry attempt counters.
- `dead_letter`: Handles persistent/max-retried errors.
- `finalize`: Normalizes and publishes a final audit/completion event.

## 5. Failure analysis

We identified and successfully handled two primary failure modes:

1. **Retry or tool failure**: Transient errors from external APIs or databases are caught in the `evaluate` node. The graph routes to `retry` which increments the attempt counter. If the number of attempts is under the threshold, it goes back to `tool`. Otherwise, it escalates to the `dead_letter` node, ensuring the system does not enter an infinite loop.
2. **Risky action without approval**: Deletions or refunds are classified as `risky` and require manual approval. They route through `risky_action` and `approval`. If approved, the action is executed. If rejected, it routes to `clarify` to request a safe alternative, preventing unauthorized operations.

## 6. Persistence / recovery evidence

The checkpointer saves checkpoints using `SqliteSaver` in the local DB. This stores state at each step, enabling human interruption and resume. For risky actions, the checkpointer acts as a crash-safe state saver.
